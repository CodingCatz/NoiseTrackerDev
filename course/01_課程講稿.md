# 課程講稿：用 AI 協作打造網頁遊戲，並上架到 GitHub Pages

> **課程定位**：零基礎也能跟著做完的「AI 協作開發」實戰課。
> 從一個遊戲點子出發，學會把它寫成一份「AI 看得懂、照著做就能完成」的規格文件，
> 再用 AI 編碼助手一步步把遊戲做出來，最後免費部署到網路上、拿到一個可以分享的網址。
>
> **最終成品**：一款 FHD（1920×1080）2D 精準平台跳躍網頁遊戲（Celeste-like，到達終點即通關），
> 用 Phaser + Vite + TypeScript 製作，透過 GitHub Actions 自動部署到 GitHub Pages。
>
> **本檔用途**：講師錄播 / 直播照稿。`【口白】` 是直接念的內容，`【操作】` 是螢幕上要做的動作，
> `【螢幕】` 是畫面提示，`【貼給 AI】` 是要交給 AI 編碼助手的指令原文。
>
> **AI 工具中立原則**：本講稿全程以「AI 編碼助手」泛稱，不綁定特定工具，方便對外發布與日後替換。

# 第 0 課　課程導論

### 0-1　開場（建立期待）

【螢幕】先播放最終成品試玩 30 秒：角色奔跑、二段跳、空中衝刺、牆跳、踩中尖刺重生、抵達終點 Clear。

【口白】
歡迎來到這門課。你現在看到的這款遊戲——可以左右跑、跳躍、二段跳、在空中衝刺、貼著牆往上爬、踩到尖刺會立刻重生、最後抵達終點通關——是我們這門課要從**完全空白**一路做到**公開上線**的成品。

而我要先講一句可能跟你預期不一樣的話：**這門課的重點，不是教你寫每一行程式。**

這門課真正要教你的，是一種新的工作方式：**如何跟 AI 編碼助手協作，把一個想法變成一個真的能跑、能分享的產品。** 你會學到怎麼把需求講清楚、怎麼把大任務拆小、怎麼驗收 AI 的產出、怎麼在它出錯時把它導回正軌。這套能力，換到任何專案、任何程式語言都適用。

### 0-2　你會得到什麼

【口白】
這門課結束時，你會擁有三樣東西。

第一，**一個真的能玩的遊戲**，跑在你自己的網址上，可以丟給朋友、放進作品集。

第二，**一段公開的開發紀錄**：你的 GitHub 上會有十幾二十次提交（commit），完整呈現這個專案怎麼一步步長出來。這在求職或接案時，比一句「我會用 AI」有說服力太多。

第三，也是最重要的——**一套可複製的 AI 協作方法論**。下一次你想做別的東西，你會知道該怎麼開頭、怎麼推進、怎麼收尾。

### 0-3　AI 協作開發的四個心法

【螢幕】四張卡片依序出現。

【口白】
在正式動手前，我要先把四個心法種進你腦袋，因為它們決定你用 AI 是「事半功倍」還是「越幫越忙」。

**心法一：任務要小。** 不要跟 AI 說「幫我做一款遊戲」，那等於沒說。要說「請只完成『跳躍時的可變跳高』這一件事，做完就停」。任務越小，AI 越不會失控，你也越容易檢查對錯。我們整門課就是把一款遊戲，拆成二十幾個小到不能再小的步驟。

**心法二：每一步都要驗收。** AI 給你程式碼，不代表它能跑。每做完一小步，我們都會跑一次建置指令（`npm run build`），用瀏覽器親眼看一次。**沒驗證過的「完成」，等於沒完成。**

**心法三：規格先行。** 我們不會邊想邊做。我們會先寫一份規格文件，把「要做什麼、數值多少、檔案怎麼擺」全部講清楚，再讓 AI 照著規格一段一段做。這就是第 1 課的主題。

**心法四：留下紀錄。** 每完成一個階段，就用 Git 提交一次。這樣萬一哪一步做壞了，你隨時可以退回上一個好的版本。AI 協作最怕的不是出錯，是**錯了卻退不回去**。

### 0-4　工具總覽（不裝太多）

【口白】
我們只需要四樣東西，而且大多是免費的：

- **一個 AI 編碼助手**——能讀你的專案檔案、能幫你改檔、能幫你跑指令的那種。
- **Node.js**——用來跑我們的開發環境，等一下我會帶你裝。
- **Git 與一個 GitHub 帳號**——免費的，用來存程式碼跟最後上架。
- **一個瀏覽器**——拿來玩我們自己做的遊戲。

【操作】示範到 Node.js 官網下載安裝、在終端機輸入 `node -v` 與 `git --version` 確認版本。

【口白】
看到版本號跳出來，就代表你準備好了。下一課，我們先不碰程式，先學一件更關鍵的事：**怎麼寫出一份讓 AI 照著做就能成功的規格。**

# 第 1 課　如何規劃一份「給 AI 執行」的規格文件

> 本課是整門課的靈魂。學生在這裡學會的不是某個遊戲的設定，而是**把任何想法轉成 AI 可執行規格的通用方法**。本課用的範例規格，就是後面 8 堂課要實作的那份。

### 1-1　為什麼不能「邊聊邊做」

【口白】
很多人用 AI 寫程式的方式是這樣的：開著對話框，想到一句講一句，「幫我加個跳躍」、「欸這個跳太高了改一下」、「啊我想加二段跳」。

一開始很爽，但通常做到一半就崩了。為什麼？因為 AI 沒有「整個專案要長什麼樣」的全局藍圖。它只看得到你最後幾句話。於是它會：改 A 弄壞 B、同一個數值在三個檔案裡寫三遍、跳躍邏輯塞進一個越來越肥、誰都看不懂的檔案。

解法只有一個：**先寫規格，再開工。** 我把它叫做「給 AI 的施工藍圖」。藍圖寫得好，AI 就是一個靠譜的施工隊；藍圖含糊，它就是一個自由發揮的災難。

### 1-2　一份好規格的七個區塊

【螢幕】拆解本課程附帶的 SPEC 文件，逐區塊對照。

【口白】
我們來解剖一份真正能用的規格。打開課程附的規格文件，你會發現它不是隨便寫的，它有清楚的骨架。我把它歸納成七個區塊，你以後寫任何規格都可以照這個骨架填。

**區塊一：專案總覽——一句話講清楚「這是什麼、不是什麼」。**
規格開頭就寫明：「這是一款精準平台跳躍遊戲，不是橫向卷軸射擊，也不是俯視角遊戲。」這個「不是什麼」極其重要。它把 AI 的想像力關進正確的籠子裡，避免它自作主張加一堆你不要的東西。

**區塊二：全域標準——所有人共用的尺。**
我們的規格定義了：畫面固定 1920×1080，而且發明了一個叫「unit」的單位，1 unit = 畫面高度的 10% = 108 像素。為什麼要這樣？因為這樣所有速度、距離都用 unit 描述，換螢幕、換解析度都不會亂。**好規格會先立一把尺，後面所有數字都用這把尺量。**

**區塊三：最終目標清單——可以打勾的驗收條件。**
規格列了十四項「完成時必須具備」的功能，從「可左右移動跳躍」到「可自動部署到 GitHub Pages」。注意，它們都是**可以打勾驗證**的，不是「手感要很好」這種摸不到的話。

**區塊四：逐項數值表——把手感變成數字。**
這是新手最容易忽略、卻最值錢的一段。「跳起來要爽」AI 做不出來，但「重力 28、跳躍高度 2.2 個單位、離開平台後 100 毫秒內還能跳」——這 AI 就能精準做出來。**規格的功力，就在於把模糊的感覺翻譯成具體的數字。**

**區塊五：檔案架構——先畫好每個東西的家。**
規格直接畫出整個資料夾長什麼樣：場景放哪、玩家放哪、物理數值放哪。這樣 AI 就不會把所有東西塞進同一個檔案。**先規定好家，東西才不會亂堆。**

**區塊六：執行總規則——AI 的行為準則。**
我們的規格有一段專門寫給 AI：不准一次重寫整個專案、每一步開始前先讀現有檔案、每一步結束前先跑建置驗收、每一步都要回報改了哪些檔案。這等於幫 AI 立家規。

**區塊七：分階段任務——把大象切成片。**
最關鍵的一段：規格把整個專案切成 Phase 0 到 Phase 21，每個 Phase 都是一段可以**單獨貼給 AI** 的任務，附明確目標、限制、跟驗收標準。我們後面 8 堂課，本質上就是一個 Phase 一個 Phase 照著貼、照著驗收。

### 1-3　「分階段任務」怎麼寫——拆解的藝術

【口白】
我特別把區塊七拿出來放大講，因為這是 AI 協作成敗的分水嶺。

看規格裡任何一個 Phase 的寫法，它永遠包含四件事：

1. **目標**——這一步要做出什麼（條列、具體）。
2. **限制**——這一步**不准**碰什麼。例如「不要實作二段跳，那留到下一階段」。
3. **預期修改**——大概會動到哪些檔案。
4. **驗收**——怎麼證明做完了，通常就是「建置成功，而且畫面上看得到某個行為」。

【螢幕】並排顯示「壞指令」vs「好指令」。

【口白】
我們對比一下。

**壞指令**：「幫我做平台跳躍的手感。」——範圍無邊無際，AI 會一次塞給你跳躍、二段跳、衝刺、牆跳，全部混在一起，錯了你根本不知道錯在哪。

**好指令**：「請只完成 Coyote Time 與 Jump Buffer。Coyote Time 設 100 毫秒、Jump Buffer 設 120 毫秒。數值要從物理設定檔讀，不要寫死。不要動到關卡跟介面。做完跑建置，並列出你改了哪些檔案。」——範圍清楚、數值明確、有護欄、有驗收。

這兩種指令，決定你是在「協作」還是在「擦屁股」。

### 1-4　怎麼從零生出這樣一份規格

【口白】
你可能會問：那這份規格本身，是怎麼來的？難道要我自己憑空寫出二十幾個 Phase？

不用。規格本身也可以跟 AI 一起生。流程是這樣：

第一步，**你先用人話講清楚願景**：我想做一款什麼類型的遊戲、核心玩法是什麼、最後要部署到哪。

第二步，**請 AI 幫你把願景擴寫成規格草稿**：列出功能清單、建議檔案架構、提出物理數值的合理範圍。

第三步，**你來做決定**：AI 提的數值、架構，你逐項確認或修改。記住，**AI 負責提供選項與草稿，拍板的人永遠是你。**

第四步，**請 AI 幫你把規格切成可執行的 Phase**：每個 Phase 小到可以獨立完成跟驗收。

我們課程附的這份規格，就是這樣生出來的。所以這堂課你真正學會的是：**不只會用規格，還會生規格。** 這是可以帶著走一輩子的能力。

### 1-5　本課小結與下一步

【口白】
這一課沒有寫半行程式，但它是整門課最值錢的一課。記住三句話：

- **先立一把尺**（全域標準），後面所有數字都用它量。
- **把感覺翻成數字**（數值表），AI 才做得出來。
- **把大象切成片**（分階段任務），每片都能單獨做、單獨驗收。

下一課，我們終於要動手了。我們會用規格的第一段任務，把這個專案的「地基」打好。

# 第 2 課　專案底座、單位制、物理資料（Phase 0–2）

### 2-1　本課目標

【口白】
這一課我們做三件事，對應規格的 Phase 0 到 Phase 2：把專案的空殼建起來、把那把「尺」（單位制）寫進程式、把遊戲的物理數值整理成一份資料。這一課做完，遊戲還不能玩，但「地基」會非常穩。

### 2-2　Phase 0：建立專案底座

【口白】
這一課每個步驟，你都**不需要懂程式術語**。你只要像在跟一個工程師朋友講需求一樣，把「要什麼、不要什麼、怎麼算成功」用白話講清楚。剩下的技術細節，AI 會對照專案規格自己補上。下面就是你可以直接照著講的版本——它一樣有目標、有限制、有驗收，這就是上一課教的「好指令」，只是換成人話。

【貼給 AI】
```txt
請幫我建立一個全新的空專案，之後要拿來做一款網頁遊戲，
技術組合請用 Vite + TypeScript + Phaser。

幾個要求：
- 遊戲畫面固定用 1920×1080 這個大小
- 把「畫面尺寸」和「1 單位 = 畫面高度的十分之一」這個換算，
  集中寫在一個設定檔，之後好統一管理
- 畫面要能自動縮放，讓電腦和手機都能完整顯示、不被裁掉
- 最後幫我確認「打包指令」可以順利跑完、沒有錯誤

這一步先把空專案和畫面設定弄好就好，先不要加任何遊戲玩法。
```

【操作】AI 改完檔後，在終端機依序執行：

```bash
npm install
npm run build
npm run dev
```

【口白】
看到瀏覽器打開、出現一片乾淨的遊戲畫面，建置沒有報錯——地基第一塊磚就放好了。

這裡停下來講一個關鍵動作：**驗收完，立刻提交。** 這就是心法四。

【操作】
```bash
git add .
git commit -m "chore(platform): phase 0 setup fhd phaser project"
```

### 2-3　Phase 1：建立場景架構

【口白】
遊戲不會只有一個畫面。會有開始選單、遊玩畫面、介面層、通關畫面。規格 Phase 1 要我們把這幾個「場景」的骨架先搭好，而且特別交代：**不要把所有邏輯塞進主檔。** 這就是規格在幫我們守住「東西不要亂堆」的原則。

【貼給 AI】
```txt
我的遊戲之後會有好幾個畫面：開機載入、開始選單、遊玩畫面、
畫面上的資訊層、還有通關畫面。請先幫我把這幾個畫面的「空架構」
搭起來，之後再慢慢往裡面填東西。

幾個要求：
- 先把上面那幾個畫面建好，可以互相切換
- 還沒有美術圖沒關係，先用簡單色塊代替，不要因為缺圖而報錯
- 一個重點：不要把所有程式都塞進同一個主檔，
  每個畫面各自一個檔案，之後才好維護

做完讓我可以看到開始選單，按 Enter 或空白鍵能進到遊玩畫面，
也順便確認打包指令能成功跑完。
```

【操作】跑 `npm run dev`，示範按 Enter 從選單進入遊戲畫面。驗收後提交。

【口白】
重點不在這幾個場景多複雜，而在於——**我們從第一天就把房間隔好了。** 玩家、介面、選單，各住各的房，後面長再大都不會打架。

### 2-4　Phase 2：物理數值資料化

【口白】
這是這一課最能體現第 1 課心法的地方。我們要把「跳多高、跑多快、重力多強」這些數字，**從程式邏輯裡抽出來，集中放在一份資料檔**。

為什麼？因為調手感的時候，你會改這些數字改上幾十次。如果它們散落在程式各處，你會找到崩潰。把它們集中在一個檔案，你調手感就像在調音響旋鈕——一個地方全部搞定。

【貼給 AI】
```txt
請幫我把遊戲的物理數值（像是跑多快、跳多高、重力多強）
從程式裡抽出來，集中放到一個專門的「設定資料檔」。

請放進這些數值：
- 最大跑速 8、最大下落速度 22
- 上升重力 50、下降重力 80（上升輕、下降重，落下更俐落）
- 起跳初速 17（大約跳 2.9 個單位高）
- 還有兩個之後調手感會用到的參數：土狼時間 100、跳躍預存 120（毫秒）

幾個要求：
- 這些數值集中在同一個檔案，不要散落在程式各處，
  更不要直接寫死在遊玩畫面裡
- 順便幫我加一個小檢查，萬一哪個數值被改成不合理的值，能提醒我

之後我要調手感，就只改這一個檔案。
```

【操作】建置驗收，提交 `feat(platform): phase 2 add physics data configs`。

### 2-5　本課小結

【口白】
這一課我們什麼都還不能玩，但做了三件影響全局的事：空殼建好、尺寫進去、數值集中。**好的開頭很無聊，但無聊的開頭換來後面的順暢。** 下一課，主角終於要動起來了。

# 第 3 課　左右移動、跳躍、Coyote Time 與 Jump Buffer（Phase 3–5）

### 3-1　本課目標

【口白】
這一課是「手感」的起點，也是整門課最能感受到「調參」樂趣的一課。我們讓角色能左右移動、能跳，然後加上兩個讓跳躍變得「好按」的小魔法：Coyote Time 跟 Jump Buffer。我刻意把三個 Phase 放一起，因為它們是同一件事——**把跳躍從『能用』調到『好用』。**

### 3-2　Phase 3：左右移動

【口白】
先讓角色會跑。注意規格的限制：「不要實作跳躍，跳躍留到下一階段。」這就是心法一——一次只做一件事。

【貼給 AI】
```txt
幫我讓遊戲裡的角色可以用方向鍵（或 A、D 鍵）左右移動。
手感不要太死：按下去是慢慢加速到全速，放開後也慢慢滑停，
不要「一按就瞬間全速、一放就瞬間定住」那種僵硬感。
這一步先把左右走做好就好，跳躍下一步再做。
```

【操作】`npm run dev`，左右移動，感受「加速、減速」的滑順感。提交。

【口白】
注意，我們要的不是「按左就瞬間全速」，而是有加速、有減速的滑順感。這個細節，就是平台遊戲手感的第一層。

### 3-3　Phase 4：跳躍與可變跳高

【口白】
現在讓他跳。這裡有個高級技巧叫「可變跳高」：輕按跳一點點，長按跳很高。幾乎所有手感好的平台遊戲都有這個。數值我們在第 2 課的設定檔裡都定好了，這裡只要用白話把需求講清楚。

【貼給 AI】
```txt
幫我讓角色可以按空白鍵（或上、W）跳起來。
我想要「輕輕點一下只跳一點點、按住不放會跳比較高」的手感。
上升時重力輕一點、往下掉時重一點，落下才俐落不飄；往下掉也不要無限加速，掉到一個最高速度就好。
數值照我們設定檔裡的來（起跳初速 17、上升重力 50、下降重力 80、最大落速 22）。
這一步先把基本跳躍做好，二段跳之後再說。
```

【操作】親自示範「輕點小跳 vs 長按高跳」的差別。提交。

### 3-4　Phase 5：Coyote Time 與 Jump Buffer（手感魔法）

【口白】
這兩個是「玩家不會發現、但拿掉就覺得卡」的隱形魔法。

**Coyote Time**（郊狼時間）：你跑出平台邊緣那一瞬間，其實已經踩空了，但遊戲偷偷給你 100 毫秒，讓你還能跳。這樣邊緣跳就不會那麼容易失敗。

**Jump Buffer**（跳躍緩衝）：你在快落地前就先按了跳，遊戲幫你記住這個按鍵 120 毫秒，一落地就自動幫你跳。這樣連續跳就很順。

【貼給 AI】
```txt
幫我把跳躍變得更好按，加上兩個容錯：
一個是「剛跑出平台邊緣那一瞬間，還能再按跳成功」（給大約 100 毫秒）；
另一個是「快落地前先按了跳，一碰到地就自動幫我跳起來」（記住大約 120 毫秒）。
這兩個數字放設定檔、不要寫死，也不要弄壞剛做好的輕按小跳、長按高跳。
```

【操作】示範：站平台邊緣「晚一點」按跳仍能跳；落地前「早一點」按跳能無縫接跳。提交。

### 3-5　遇到手感不對怎麼辦——除錯指令範本

【口白】
調手感一定會遇到「怪怪的」。這時不要慌、不要叫 AI 大改。用規格附的這個範本，**精準描述問題、框死範圍**：

【貼給 AI】
```txt
我覺得跳躍手感怪怪的：
輕按和長按跳起來差不多高、往下掉太慢、站在平台邊緣晚一點按就跳不動。
請只調這幾個手感的地方，不要動到關卡和畫面；
而且這些規則要保留：起跳初速 17、上升重力 50、下降重力 80、土狼時間 100、跳躍預存 120。
```

【口白】
看到沒？**講清楚哪裡怪、只改這一塊、這些別動、這些規則要留著。** 這套「除錯四件組」，你後面會一直用到。

# 第 4 課　二段跳、Dash、牆滑與牆跳（Phase 6–8）

### 4-1　本課目標

【口白】
這一課我們給角色三個「招式」：在空中再跳一次的二段跳、瞬間衝刺的 Dash、貼著牆往上爬的牆滑牆跳。做完這課，你的角色就有了一個真正平台遊戲該有的全套身手。

這一課也會第一次出現「能力系統」這個概念——我們不是把這些招式硬塞進角色，而是做一個專門管理「角色解鎖了哪些能力」的系統。這為第 7 課的「能力門檻關卡設計」鋪路。

### 4-2　Phase 6：二段跳

【貼給 AI】
```txt
幫我加一個二段跳：角色跳到空中後，還能再按一次跳、再往上跳一段。
落地之後就重置，下次又能再用一次。
請把招式的邏輯集中在一個地方管理，不要散在角色程式裡（之後招式會越來越多）。
畫面上顯示角色目前會哪些招式，方便我測試。
這一步先做二段跳，衝刺下一步再做。
```

【口白】
注意我們把「招式邏輯集中管理」當成一件事來做——招式之後會越來越多，先把管理的地方蓋好，後面才不會亂。能力本身一開始就都有，關卡的難度之後靠地形設計和鑰匙、門來控制，不是靠鎖能力。先把招式做到能動，手感細節之後再慢慢調對：**先讓它動，再讓它對。**

### 4-3　Phase 7：空中衝刺 Dash

【貼給 AI】
```txt
幫我加一個空中衝刺：按 Shift（或 L、C），角色會瞬間往前衝一小段。
衝刺距離大約 3 個單位、很短一下子（約 0.15 秒）。
有按方向就往那個方向衝，沒按就朝目前面對的方向衝。
不能因為衝刺就穿過牆壁；落地後重置，空中只能衝一次。
做完顯示「衝刺：可用」，別弄壞跳躍和二段跳。
```

【操作】示範空中衝刺穿越一段水平缺口。提交。

### 4-4　Phase 8：牆滑與牆跳

【口白】
最後一招，也是最容易做出 bug 的一招。牆跳常見的毛病是：跳開牆面後，玩家一推方向鍵就「黏」回牆上。規格早就預見了這點，所以有個 `wallJumpLockMs = 120`——牆跳後 120 毫秒內鎖住水平輸入，讓玩家先離開牆面。

【貼給 AI】
```txt
幫我加牆滑和牆跳：
角色在空中貼著牆往下掉時，會變成慢慢沿著牆滑下去、不要掉太快；
這時候按跳，就往反方向的斜上方跳開。
跳開後短時間內（約 120 毫秒）先別讓方向鍵把人拉回牆上，不然會一直黏在牆邊。
注意別把地板誤判成牆，也別弄壞二段跳和衝刺。
```

【口白】
如果牆跳真的「黏牆」了，用這個對症下藥的指令：

【貼給 AI】
```txt
牆跳之後角色很容易立刻又黏回牆上，請修。
正確的感覺是：牆跳會把人往反方向推開，而且推開後那 120 毫秒內，
就算我按著貼牆方向，也不會馬上被拉回去。
另外，牆滑只能在「空中貼牆」時發生，站在地上不該觸發。
請只調牆滑牆跳這塊，別動到關卡和畫面。
```

【口白】
這一課的大收穫：**越複雜的功能，越要靠規格預先想到的「護欄數值」。** 牆跳那個 120 毫秒，就是規格作者替你踩過的坑。

# 第 5 課　Tilemap 關卡與 Camera（Phase 9–10）

### 5-1　本課目標

【口白】
前四課，我們的角色一直在一片空地上練功。這一課，我們給他一個真正的「世界」——有高有低的平台、需要二段跳才上得去的高台、需要 Dash 才跨得過的缺口、需要牆跳才爬得上的垂直通道。然後我們給鏡頭裝上「跟拍」，讓它跟著角色跑、但不會跑出世界邊界。

### 5-2　Phase 9：關卡資料架構

【口白】
這裡又一次體現「資料驅動」的威力。我們**不會**把平台座標一個一個寫死在程式裡。我們會把整個關卡——哪裡有平台、哪裡有缺口、起點終點在哪——寫成一份「關卡資料」，再讓一個 LevelSystem 照著資料把世界蓋出來。這樣以後要改關卡，改資料就好，不用碰程式。

【貼給 AI】
```txt
幫我蓋一個可以實際玩的關卡，而且要「資料驅動」——
把哪裡有平台、哪裡有缺口、起點終點在哪，寫成一份「關卡資料」，
再由程式照著它把場景蓋出來。
不要把平台座標一個一個寫死在遊玩畫面裡，之後我才好改關卡。
這個關卡至少要有：起點、高低不同的平台、一段要二段跳的高台、
一段要衝刺的缺口、一段要牆跳的垂直通道、還有終點。
```

【口白】
你可能會問：AI 寫出來的「關卡資料」到底長什麼樣？其實它就是**兩張清單**，一點都不可怕。

把關卡想成一張**方格紙**，採 Phaser 原生座標：橫向叫 x、縱向叫 y，而且 **`(x, y)` 指的是物件的左上角、`y` 越大越靠下**（不翻轉，整個專案只用一套座標系）。每個平台、機關都用「左上角 x、y ＋ 寬 w、高 h」這四個數字描述；出生點是一個點（只有 x、y）。座標單位都是「單位 (U)」，載入時統一 ×108 變像素。

```txt
出生點 = { x: 1, y: 6 }
平台 = [
  { x: 0,  y: 7, w: 8, h: 2 }   # 地面
  { x: 11, y: 3, w: 3, h: 1 }   # 高台（要二段跳才上得去）
]
終點 = { x: 13, y: 5, w: 1, h: 2 }
```

學生不用懂程式，只要會**填這些清單**：想搬平台就改 x、y，想改大小就改 w、h，想加機關就多寫一行。這就是「資料驅動」對新手真正的好處。

【口白】
而且這個關卡是**刻意設計**的：每一段地形都對應一個我們前面做好的能力。這就是平台遊戲的精髓——**用關卡逼玩家用上每一個招式。**

### 5-3　Phase 10：Camera 跟拍

【貼給 AI】
```txt
幫我讓鏡頭跟著角色跑。
跟拍要平滑一點，不要硬邦邦地黏著角色、晃來晃去。
鏡頭不可以拍到關卡外面——走到邊緣時，鏡頭就停在世界邊界。
另外，角色死掉或重重落地時，給鏡頭一點輕微震動，增加打擊感。
```

【操作】示範鏡頭平滑跟拍、走到關卡邊緣時鏡頭停在邊界、死亡時輕微震動。提交。

### 5-4　本課小結

【口白】
這一課之後，你的東西第一次像「一個遊戲」了：有世界、有鏡頭、有可以挑戰的地形。下一課，我們加入讓世界「活起來」的互動：鑰匙、門、開關、會殺死你的尖刺、會復活你的存檔點。

# 第 6 課　鑰匙門開關、Checkpoint、陷阱與移動平台（Phase 11–13）

### 6-1　本課目標

【口白】
這一課是「關卡機關大禮包」。我們加三種東西：可以撿、可以開門的鑰匙與門；踩到會死、會把你送回存檔點的尖刺與深淵；以及記錄你進度的存檔點（Checkpoint）。做完這課，關卡就有了「玩」的內容，而不只是「跑」的地形。（載著你移動的平台依規格的範圍界定先不做，列為進階選配，這一課最後會說明。）

### 6-2　Phase 11：鑰匙、門、開關

【貼給 AI】
```txt
幫我加遊戲裡的互動機關：鑰匙、門、開關。
角色碰到鑰匙就撿起來，畫面上顯示目前有幾把。
有些門要有鑰匙才能開；另做一種開關，踩一下會打開對應的門。

這些機關放哪、誰開誰，請寫進關卡資料，不要寫死在程式裡。
```

### 6-3　Phase 12：Checkpoint 與重生

【口白】
這是精準平台遊戲的靈魂機制。我們的遊戲沒有血量、不會「Game Over 淘汰你」——你踩到尖刺、掉進深淵，就**立刻重生在最近的存檔點**，再挑戰一次。死亡不是懲罰，是鼓勵你「再試一次」。這正是 Celeste 那類遊戲讓人上癮的原因。

【貼給 AI】
```txt
幫我加存檔點和死亡重生。
角色碰到存檔點就啟用它，並亮起來讓我看得出來。
這個遊戲沒有血量——踩到尖刺、或掉進深淵，就立刻重生回最近啟用的存檔點。
重生後速度歸零、回到站好的狀態，不要一重生又馬上死。
```

【操作】示範：啟用存檔點 → 故意踩尖刺 → 瞬間重生回存檔點。驗收「重生後不會立刻又死」。提交。

### 6-4　Phase 13：陷阱

【貼給 AI】
```txt
幫我加陷阱：尖刺和深淵。
角色碰到尖刺、或掉進深淵，就立刻死亡、重生回最近的存檔點。
深淵＝掉出關卡下方就算死，不要讓角色無限往下掉。
關卡裡至少放一處尖刺、一段會掉下去的深淵。
```

【口白】
這裡補一句「不做什麼」：載著你移動的平台，規格把它列為**進階選配**，這一版先不做。原因是 YAGNI——先把核心做穩。但我們留好接口：之後要加，最乾淨的做法是把它當成「開關的目標之一」（踩開關平台才動），第 7 課的開關連動設計能直接接上，不用打掉重練。

到這裡，機關越加越多，「誰碰到誰、然後怎樣」的判斷已經散在好幾個檔案了——這就是下一課要做「碰撞系統重構」的最好理由：當判斷開始散落、難維護，就是該把它獨立出來好好整理的訊號。

# 第 7 課　CollisionSystem 重構、能力門檻、通關流程（Phase 14–16）

### 7-1　本課目標

【口白】
前面六課，我們一直在「加東西」。這一課，我們第一次做「整理」——這在真實開發裡叫「重構」（refactoring）。我們要把散在各處的碰撞邏輯收進一個專門的系統；用地形和鑰匙、門設計關卡的推進門檻；最後把整個遊戲的通關流程串起來。

我特別重視這一課，因為**會加功能的人很多，會整理的人很少。** 而 AI 協作時代，「整理」恰恰是 AI 最能幫上忙、人類最該主導的地方。

### 7-2　Phase 14：抽出 CollisionSystem（重構）

【口白】
到目前為止，「誰碰到誰、然後怎樣」這種判斷，可能散落在好幾個檔案。我們要把它們全部收進一個叫 CollisionSystem 的地方統一管理。重構最重要的紀律是：**功能不能退化。** 整理完，遊戲玩起來要跟整理前**一模一樣**，只是程式更乾淨。

【貼給 AI】
```txt
到現在，「誰碰到誰、然後怎樣」這種判斷，散在好幾個檔案裡。
幫我把它們全部收進一個專門管碰撞的地方統一處理：
角色踩地板、踩到尖刺死亡、掉進深淵、撿鑰匙/道具、開門踩開關，都集中在這裡判斷。
最重要的一點：這是「整理」，不是改玩法——
整理完，遊戲玩起來要跟現在一模一樣，不能有任何功能變不一樣或壞掉。
```

【口白】
驗收這一步的方法很特別：**你要把前面所有功能再玩一遍**，確認沒有任何東西壞掉。重構的驗收標準是「什麼都沒變」——這聽起來很反直覺，但這正是好重構的定義。

### 7-3　Phase 15：關卡推進門檻

【口白】
這一課的另一塊，是設計關卡的「推進門檻」。注意：精準平台的卡關，不是靠鎖住能力，而是靠**地形難度**和**鑰匙、門**。能力玩家一開始就都有；難在哪？難在那段高台非二段跳上不去、那個缺口非衝刺跨不過、那條垂直通道非牆跳爬不上，外加有些路要先找到鑰匙開門。**用設計卡關，不用鎖能力。**

【貼給 AI】
```txt
幫我用地形難度和鑰匙、門，設計關卡的推進門檻。
地形門檻：高台要二段跳、缺口要衝刺、垂直通道要牆跳，逼玩家用對招式才過得去。
鑰匙門門檻：有些路要先找到鑰匙開門、或踩開關才走得通。
能力本身一開始就都有、不用解鎖——關卡靠設計來控制難度。
```

### 7-4　Phase 16：GameState 與通關流程

【貼給 AI】
```txt
幫我把整個遊戲的進度狀態集中管理：
死了幾次、拿了幾把鑰匙、解鎖了哪些能力、目前在哪個存檔點、有沒有通關。
角色抵達終點就進到「通關畫面」，顯示 Clear（可順便顯示死亡次數）。
這個遊戲沒有 Game Over，死掉只會重生、不會被淘汰。
通關畫面可以按 R 重新開始；重來時，記得把上一輪的舊狀態和舊的事件監聽清乾淨，
不然玩第二輪會出現怪問題。
```

【口白】
注意最後那個「重新開始時要清除舊的 listener」——這是 AI 協作裡很容易忽略、卻會造成「玩第二輪就出怪事」的隱藏 bug。規格幫我們點出來了，這就是好規格的價值。這一課做完，你的遊戲已經是一個**有頭有尾、可以從開始玩到通關、再重來**的完整循環了。

# 第 8 課　Debug overlay、視覺回饋、Mobile 控制（Phase 17–19）

### 8-1　本課目標

【口白】
這一課做三件「讓遊戲更好調、更好看、更多人能玩」的事：一個能即時顯示物理數據的除錯面板、一批讓各種狀態看得出差別的視覺回饋、以及讓手機也能玩的觸控按鈕。

### 8-2　Phase 17：Debug Overlay（調手感的神器）

【口白】
這個面板，是這整門課「調手感」最強的武器。按 F3，畫面就會即時顯示：角色現在的座標、速度、是不是踩在地上、是不是貼著牆、Coyote 計時器還剩多少……

為什麼重要？因為手感問題**用眼睛猜很難，用數據看很準**。當你覺得「跳躍怪怪的」，打開這個面板，你會直接看到「喔，下落速度衝到 20 了，難怪太快」。**把看不見的物理狀態變成看得見的數字——這是專業開發者除錯的核心習慣。**

【貼給 AI】
```txt
幫我做一個除錯面板，按 F3 可以開關（預設關閉）。
打開時，在畫面角落即時顯示角色現在的狀態：
座標、左右和上下的速度、有沒有踩在地上、有沒有貼著牆、
土狼時間還剩多少、跳躍預存還剩多少、目前是哪個狀態、還有 FPS。
這是我調手感用的工具，不要影響正常遊玩，也不要為它裝很重的外掛。
```

### 8-3　Phase 18：視覺回饋

【貼給 AI】
```txt
幫我加一些「看得出差別」的視覺回饋：
角色在站、跑、跳、下落、衝刺、貼牆滑時，外觀要有簡單的差別；
死亡時閃爍或碎裂一下、重生時淡入；
碰到存檔點會亮起、撿到能力道具有光效、開門和踩開關也有反應。
還沒有美術圖沒關係，先用簡單的色塊和形狀代替就好，重點是把「回饋」做出來。
```

【口白】
這裡用的全是「佔位圖」（placeholder）——簡單的色塊、閃爍、淡入。我們不需要美術素材也能做出「狀態看得出差別」的回饋。**先把回饋的機制做對，美術可以之後再換。**

### 8-4　Phase 19：Mobile 觸控（選配）

【口白】
先講清楚定位：規格是**以鍵盤為主**、未來可能加手把，手機觸控屬於「讓更多人能玩」的額外加分，不是必做。所以這一段做與不做都可以；要做的話，關鍵是別動到鍵盤體驗。

【貼給 AI】
```txt
幫我加手機觸控操作（這是選配）：
在手機或平板上，畫面出現虛擬按鈕，可以左右移動、跳、衝刺、互動。
電腦上的鍵盤操作完全不變。
這些觸控按鈕只在「手機/觸控、或小螢幕」時才出現，而且不要擋住太多遊戲畫面。
```

【口白】
注意限制裡那句「桌機鍵盤操作維持不變」——加新功能時，**最重要的事是別弄壞舊功能**。這又是一次「不退化」的紀律。

# 第 9 課　上架 GitHub Pages 與課程交付（Phase 20–21）

### 9-1　本課目標

【口白】
最後一課，我們做兩件讓這個專案「對外存在」的事：設定自動部署，讓你每次推程式碼上去，網站就自動更新；以及整理交付文件，讓任何人（包括未來的你）一打開專案就知道這是什麼、怎麼跑、怎麼玩。

這一課做完，你就有一個**公開的網址**了。

### 9-2　Phase 20：GitHub Pages 自動部署

【口白】
我們要做的是「自動部署」：你把程式碼推上 GitHub，GitHub 會自己幫你建置、自己幫你上線。這套機制叫 GitHub Actions。規格已經把整個設定檔寫好給我們了。

【貼給 AI】
```txt
幫我設定自動部署到 GitHub Pages：
之後我只要把程式碼推上 GitHub，網站就自動重新打包、自動更新上線，我不用手動做任何事。
請用 GitHub 官方的 Pages 功能來做。
另外注意：GitHub Pages 的網址會多一層專案名稱的路徑，
請幫我把專案設定調對，免得上線後變成白畫面、圖都載不出來。
```

【螢幕】展示 `deploy.yml` 與 `vite.config.ts` 的 base path 設定（內容見規格 Phase 20）。

【口白】
這裡有個新手一定會踩的坑：GitHub Pages 的網址會多一層專案名稱的路徑。所以 `vite.config.ts` 的 `base` 一定要設對，不然遊戲上線後會「白畫面、圖都載不出來」。規格裡那段自動抓 repo 名稱的設定，就是在替你避開這個坑。

### 9-3　實際上線：把開關打開

【操作】完整示範一次：
```bash
git add .
git commit -m "ci(platform): phase 20 deploy to github pages"
git push
```
然後到 GitHub 倉庫 → Settings → Pages → Source 選 **GitHub Actions** → 回到 Actions 分頁看部署流程跑完 → 點開網址。

【口白】
看到那個綠色的勾，再點開網址——你的遊戲，現在全世界都能玩了。把這個網址記下來，等一下要寫進 README。

### 9-4　Phase 21：課程交付文件

【貼給 AI】
```txt
幫我整理這個專案的交付文件，給看的人（包括未來的我）一打開就懂：
更新 README，寫清楚這是什麼遊戲、怎麼操作、有哪些能力、怎麼開發、怎麼部署、還有遊戲網址；
另外建立三份紀錄表：一份記錄我下過哪些指令、一份記錄試玩測試、一份記錄調手感的過程。
請照專案目前「真實的樣子」寫，不要寫成廣告、也不要假裝做了還沒做的功能。
```

【口白】
最後這份 `ai-prompts-log.md` 特別有意思——它讓你**回頭記錄整個專案下過哪些指令**。這不只是作業，它是你這趟 AI 協作旅程的航海日誌。下次做新專案，翻一翻就知道哪種指令最有效。

### 9-5　課程總收尾

【螢幕】回放第 0 課的最終成品試玩，這次旁邊疊上「這是你做的」。

【口白】
我們從一份規格出發，走過二十幾個小步驟，現在你手上有：一個能玩的遊戲、一個公開網址、一段完整的 GitHub 開發紀錄。

但我希望你帶走的不是這款遊戲。我希望你帶走的是這套方法——

**先寫規格、把大象切成片、一次只做一件事、每步都驗收、隨時能退回。**

這套方法跟「平台遊戲」一點關係都沒有。它是你未來跟 AI 一起做**任何東西**的底層作業系統。今天你用它做了一款遊戲；明天，你可以用它做任何你想做的東西。

恭喜你完成這門課。去把你的網址分享出去吧。
