專案底座、單位制、
物理資料

三件影響全局的事：把空殼建起來、把那把「尺」寫進程式、把物理數值整理成一份資料。

這一課做完，
遊戲還不能玩，
但地基會非常穩。

本課做三件事

把地基打好

  • 建空殼　最小可運行的 Vite + TypeScript + Phaser 專案
  • 把尺寫進程式　1920×1080 與 1 unit = 108 px 的單位制
  • 物理數值資料化　跳多高、跑多快、重力多強，集中成一份資料

PHASE 00

建立專案底座

把規格的第一段任務，原封不動貼給 AI——有目標、有限制、有驗收。

▍你不用懂術語，像跟工程師朋友講需求一樣說就好

💬 你對 AI 說

請幫我建立一個全新的空專案，之後要拿來做一款網頁遊戲，技術組合用 Vite + TypeScript + Phaser。

遊戲畫面固定用 1920×1080 這個大小 把「畫面尺寸」和「1 單位＝畫面高度的十分之一」這個換算，集中寫在一個設定檔，之後好統一管理 畫面要能自動縮放，讓電腦和手機都能完整顯示、不被裁掉 最後幫我確認「打包指令」可以順利跑完、沒有錯誤

這一步先把空專案和畫面設定弄好就好，還不要加任何遊戲玩法。

每一步的標準收尾

先確認沒壞，再存一個版本

💬 你對 AI 說（也可以自己做）

幫我打包一次、確認沒有任何錯誤；沒問題的話，把目前的進度存成一個版本，註明這是「建好專案底座」。

背後其實就是這兩行，之後每一課都會用到：

npm run build　　# 打包，確認沒有錯
git commit -m "建好專案底座"　　# 把進度存成一個版本

PHASE 01

建立場景架構

遊戲不只一個畫面。從第一天，就把每個房間隔好。

五個場景骨架

各住各的房，長再大都不打架

  • BootScene　產生 placeholder 圖，避免缺圖錯誤
  • MenuScene　標題與開始提示
  • GameScene　遊玩主畫面（暫時放底色與 placeholder）
  • UIScene / GameOverScene　介面層 / 通關畫面

鐵則：不要把所有邏輯塞進 main.ts。

PHASE 02

物理數值資料化

把「跳多高、跑多快、重力多強」從程式邏輯抽出來，集中成一份資料。

Phase 02 · 你要 AI 做的事

把數字集中到一個檔案

💬 你對 AI 說

幫我把遊戲的物理數字（跑多快、跳多高、重力多強）集中放到一個專門的設定檔，之後我要調手感，就只改這一個檔案。

AI 做出來的檔案大概長這樣——你看得懂這些數字就夠了：

物理數值 = {
  最大跑速  8　　　上升重力  50
  最大下落  22　　 下降重力  80
  起跳初速  17　　 跳躍高度  ≈2.9
  土狼時間  100　 跳躍預存  120　（毫秒）
}

數值集中一處，
調手感就像調音響旋鈕——
一個地方，全部搞定。

本課小結

好的開頭很無聊，
但換來後面的順暢。

空殼建好、尺寫進去、數值集中——這一課什麼都還不能玩，卻決定了整個專案能不能順順地長大。下一課，主角終於要動起來。
